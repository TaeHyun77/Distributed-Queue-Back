### 기존 단일 대기열 시스템에서의 변경점

---

기존 대기열 시스템에서는 특정 대기열에서 변동 사항이 발생할 때, DB 트랜잭션과 이벤트 전송의 정합성을 보장하기 위해 MySQL과 Kafka Debezium Connector를 활용한 CDC 방식으로 이벤트를 전달했습니다.

하지만, 이 방식은 데이터베이스 의존도가 과도했으며, 대기열 상태 관리 자체에 DB가 반드시 필요하지 않았고, DB가 단일 장애점이 될 가능성이 존재했기에 이를 제거하였습니다.

또한, Docker를 활용하여 단일 서버를 분산 환경으로 확장함으로써 가용성을 높였습니다.<br><br>

### 분산 환경에서 발생한 문제점 및 추가 기능 구현

---

1. 서버 인스턴스별 독립적인 MutableSharedFlow로 인한 SSE 이벤트 전달 누락 문제 💣

2. 분산 환경에서의 스케줄링 중복 실행 문제 💣

3. Redis Sentinel 기반 Replica 구조를 구성하여 고가용성을 확보하였습니다.

4. k6 부하 테스트로 RPS를 단계적으로 증가시키며 시스템 처리 한계를 측정하였습니다.<br><br>

### Kafka 도입과 제거

---

**도입 배경**

티켓팅 서비스 특성상 공연 오픈 시각에 수만 명이 동시에 몰리는 상황을 전제로 설계했습니다.

- 목표 처리량 : 1,000 RPS
- 
- 목표 지연 시간 : E2E 1초 이하<br><br>

분산 환경에서 여러 인스턴스가 동시에 요청을 처리할 때 순서 정합성 보장, 서버 장애 시 메시지 유실 방지, 급격한 트래픽 급증 시 부하 완충이라는 세 가지 목적으로 Kafka를 도입하였습니다.<br><br>

초기 아키텍처는 아래와 같습니다.
```
사용자 → Nginx → Spring WebFlux (3 인스턴스) → Kafka → Consumer → Redis
```
<br>

**벤치마크 테스트 결과 (Kafka 적용)**

| RPS | avg (ms) | p95 (ms) | E2E p95 (ms) | E2E max (ms) | 처리량 (/s) |
| --- | --- | --- | --- | --- | --- |
| 100 | 34.23 | 59.08 | 67 | 136 | 74.1 |
| 200 | 48.03 | 101.23 | 197 | 395 | 154.0 |
| 300 | 44.42 | 95.56 | 190 | 378 | 230.9 |
| 400 | 74.44 | 158.97 | 346 | 589 | 307.8 |
| 500 | 111.81 | 278.52 | 544 | 842 | 400.0 |
| 600 | 249.94 | 773.41 | 1,197 | 1,977 | 461.8 |

<br><br>

모든 구간에서 성공률은 100%였지만 RPS 600에서 E2E p95가 1.2초를 초과하며 목표였던 1초 이하를 달성하지 못했습니다. 또한 RPS가 증가할수록 HTTP 응답 시간과 E2E 시간의 격차가 비선형적으로 확대되었습니다.<br><br>

**원인 파악**

Consumer의 배치 처리(최대 200건)로 Consumer Lag은 크게 발생하지 않았습니다. 즉, 처리 속도 문제가 아닌 Kafka를 경유하는 구조 자체가 만드는 지연이 근본 원인이었습니다.

`acks=all` 설정으로 인해 produce 단계에서 3개 브로커의 복제 완료까지 대기하는 지연이 발생하고, produce 완료 후에도 Consumer가 다음 poll을 수행하기까지의 비동기 간극이 추가됩니다. RPS가 올라갈수록 produce 단계의 경합이 심해지면서 이 지연은 비선형적으로 증가했습니다.<br><br>

**도입 이유 재검증**

- **영속성** : 실제 상태는 Redis에 저장되므로 Redis 장애 시 Kafka 재처리도 실패하는 구조여서 영속성의 실효성이 낮았습니다.

- **재시도** : Redis Sentinel failover 시간에 맞춰 재시도 간격을 늘리면 해당 파티션의 후속 메시지 처리가 전부 멈추는 문제가 발생하여, 실시간 대기열 시스템에서는 치명적이었습니다.

- **부하 완충** : 1,000 RPS 기준 Redis에 발생하는 연산량은 Redis 최대 처리량의 약 9% 수준에 불과하여 완충이 필요한 상황 자체가 아니었습니다.<br><br>

결과적으로 Kafka는 확장성을 만든 것이 아니라 지연과 복잡도를 만드는 계층이 되었습니다.<br><br>

**제거 후 개선**
```
사용자 → Nginx → Spring WebFlux (3 인스턴스) → Redis → 즉시 처리 → 응답
```
<br>

**기본 설정**

Docker를 활용하여 분산 환경을 구성하였으며, 각 애플리케이션 인스턴스는 시작 시 Redis Pub/Sub 채널을 구독하여 이벤트를 수신할 준비를 합니다.<br><br>

**분산 서버에서의 대기열 등록 과정**<br>

1. 클라이언트는 대기열 등록 요청 시 대기열 식별자인 queueType, 사용자 식별자인 userId를 서버로 전달합니다.

    서버는 요청 도착 시각을 timestamp로 기록하며, 클라이언트와 SSE 연결을 수립합니다.

2. Redis SET NX 명령어로 중복 요청을 검증한 뒤, userId를 member로, timestamp 기반의 score를 Redis Sorted Set에 삽입합니다. zadd 명령어의 원자성에 의해 대기열 순서가 요청 시각 기준으로 자동 정렬됩니다.

3. Redis Pub/Sub을 통해 대기열 갱신 이벤트를 모든 서버 인스턴스에 전파하여, 각 서버가 동일한 대기열 상태를 공유할 수 있도록 합니다.

    클라이언트는 SSE 연결을 통해 대기열 상태가 변경될 때마다 실시간으로 갱신된 정보를 수신합니다.

4. 대기열 시스템은 대기열과 참가열 두 영역으로 구성되며, 스케줄러가 주기적으로 대기열에서 일정 인원을 참가열로 승격시킵니다. 참가열로 승격된 사용자는 타겟 페이지에 접근할 수 있는 권한을 획득하며, 클라이언트는 해당 페이지로 이동합니다.

5. 참가열로 승격된 사용자에게는 queueType과 userId 기반의 접근 토큰을 생성하여 클라이언트 쿠키에 저장합니다. 이후 사용자가 타겟 페이지에 진입하면, 서버는 동일한 방식으로 토큰을 재생성하여 클라이언트가 전달한 토큰과 비교하고, 참가열에서의 TTL 만료 여부를 함께 확인하여 사용자의 접근 권한을 검증합니다.<br><br>


**대기열과 참가열 분리**

이번 대기열 프로젝트에서는 대기열과 참가열을 분리하여 구성하였습니다.

만약 대기열 하나로만 입장을 관리한다면, 순서가 된 사용자들을 대기열에서 제거하는 즉시 해당 사용자는 시스템 어디에도 존재하지 않게 됩니다.

이 상태에서 페이지 새로고침이나 네트워크 끊김이 발생하면, 서버는 이 사용자가 입장 권한을 가진 사용자인지 판별할 수 없으며, 현재 서비스를 이용 중인 사용자 수를 파악하기 어려워 동시 접속 제한도 까다로워집니다.

이를 해결하기 위해 대기열과 참가열로 두 개의 큐를 구성하였습니다. 대기열은 사용자가 진입하여 순위를 확인하며 입장을 기다리는 큐이고, 참가열은 스케줄러에 의해 승격된, 입장이 확정된 사용자가 머무는 큐입니다.

이렇게 분리하면 참가열의 크기가 곧 현재 이용 중인 사용자 수이므로 동시 접속 제어가 단순해지고, 참가열 토큰에 만료 시간( TTL )을 부여하여 이탈한 사용자의 자리를 자동으로 회수할 수 있으며, 각 큐가 순서 관리와 입장 권한 관리라는 명확한 책임을 가지게 되어 유지보수도 용이해집니다.<br><br>

**승격 로직**

승격 로직은 대기열의 사용자를 참가열로 이동시키는 로직입니다. 참가열의 최대 허용 인원은 예약 시스템이 안정적으로 처리할 수 있는 부하 수준을 기준으로 설정해 두었습니다. 

사용자가 대기열에 등록할 때 참가열에 여유가 있다면 바로 참가열로 삽입하고, 여유가 없다면 대기열에 삽입합니다. 

이후 스케줄러가 3초 주기로 참가열의 잔여 수용량을 계산하여, 여유가 생긴 만큼 대기열의 사용자를 순서대로 참가열로 이동시킵니다.
